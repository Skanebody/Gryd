## Contradictions Spéc ↔ existant (arbitrées)

- **Onboarding : 'arrivée directe sur carte démo explorable' (§7.2)**  
  Conflit : AMENDEMENT-47 (CONTRAIGNANT) a SUPPRIMÉ le mode démo : données réelles ou vides, jamais 'de démonstration' ; et la carte exige une session dès qu'un backend existe (Redirect /sign-in), donc pas d'exploration anonyme.  
  ⚖️ Arbitrage : AMENDEMENT-47 prime. Réécrire §7.2 : 'carte réelle ou vide + compte requis' au lieu de 'données de démonstration'. La lecture anonyme de zones publiques agrégées reste possible comme chantier backend optionnel — jamais en fabriquant des données.
- **Microcopy du CTA central : 'GO' (§20)**  
  Conflit : Le code peint un verbe contextuel (nav actionRun='RUN', deriveContextualAction run/defend/conquer) et la skill 'zéro GO' veut des verbes contextuels ; or l'AMENDEMENT-38 (override fondateur) a déjà tranché GO.  
  ⚖️ Arbitrage : GO gagne : §20 CONFIRME l'override A-38. Réconcilier le libellé 'RUN' résiduel vers 'GO' sur le CTA central, garder l'intention contextuelle (Conquérir/Défendre) comme sous-texte, pas comme libellé du bouton. La skill 'zéro GO' est supplantée.
- **i18n : §23 exige ICU MessageFormat**  
  Conflit : Le repo n'a PAS de locales/*.json : i18n = catalogue TypeScript typé (Entry, parité 5 langues imposée par le type) avec pluriels MANUELS (paires one/many + branchement v>1). format() ne fait que de l'interpolation {var}.  
  ⚖️ Arbitrage : Garder le catalogue TS typé (supérieur : parité garantie à la compilation, impossible avec des JSON). Ajouter la sémantique ICU (Intl.PluralRules / plural-select-number) DANS format() ; ne pas adopter le format de fichier locales/*.json que la lettre de §23 semble impliquer.
- **Onboarding humain : §13.3 exige une photo/vidéo d'un runner/cycliste réel**  
  Conflit : Le code montre une démonstration SVG purement géométrique (chip 'Exemple') — l'humain est totalement absent ; §13.4 et AMENDEMENT-47 interdisent la photo stock.  
  ⚖️ Arbitrage : Décision fondateur (voir décisions). Deux issues honnêtes seulement : vraie photo non-stock (diversité, gradient carbone) OU acter la démo géométrique et lever §13.3. Fabriquer/acheter du stock est exclu par §13.4 + A-47.
- **Vélo comme discipline de jeu first-class (Partie II §2/§5/§8)**  
  Conflit : Le code grave le vélo comme TRICHE : validation.ts reason 'pace_too_fast (anti-vélo)', strava_import rejette 'Ride', appleHealth ne lit que .running, admin/simulate classe 'bike' en CheatMode 'doit être rejetée' — le tout protégé par des tests.  
  ⚖️ Arbitrage : La vision 'vélo=triche' est OBSOLETE selon la nouvelle spec, mais à remplacer CHIRURGICALEMENT, pas globalement : paramétrer par discipline (RUN_*/BIKE_*), garder l'invariant pour le mode course, accepter 'Ride'/.cycling seulement quand FEATURE_BIKE_MODE est on. Tout derrière flag, post-stabilité Run (§5.3).
- **Classements atteignables visibles (§16, esprit §2.6)**  
  Conflit : flags.season = FULL_SURFACE (défaut OFF) masque tout l'onglet Saison/classement en MVP (décision de scope D8).  
  ⚖️ Arbitrage : Décision fondateur (voir décisions). Si l'atteignable §2.6 prime, lever flags.season pour la portée locale/quartier au minimum ; sinon inscrire §16 'post-MVP' dans la spec. Les scores s'accumulent déjà serveur (ré-affichables d'un flip) — pas de donnée fabriquée d'ici là.
- **'Remplacer le niveau générique' par la progression statutaire (§12.1)**

## Détail par domaine

### ONBOARDING (§7 parcours + §8 permissions + §13.3 photo onboarding)

L'onboarding est le domaine le plus abouti de l'app et il est LARGEMENT conforme à la spec §7/§8 : parcours linéaire de 4 écrans (mechanic → rivalry → city → account), verrouillé par test Deno (17/17 verts), faisable assis/dans le métro/sans GPS, avec un raccourci de position FACULTATIF portant sa pré-permission affichée en permanence. Aucune permission n'est demandée par le flow (ni notif, ni ATT — absente de tout le repo mobile, ni santé/montre, ni photothèque), aucun paywall, aucun choix de sport/couleur/crew : §7.3 est respecté point par point, et le GPS de course est repoussé au premier GO (§7.2 « première course »). La copie de l'écran 1 est mot pour mot celle de la spec (« Ferme une boucle. Prends la zone. »). Deux vrais écarts subsistent, tous deux ASSUMÉS et documentés dans le code : (1) §13.3 exige une PHOTO/vidéo d'un runner/cycliste réel sur l'écran 1 — le code montre à la place une démonstration SVG géométrique abstraite (chip « Exemple »), l'humain est totalement absent de l'onboarding ; (2) §7.2 « arrivée directe sur carte démo » n'est que partielle : pas de permission GPS forcée (conforme) mais la carte exige une SESSION dès qu'un backend est branché, donc pas d'exploration avant compte, et la carte est réelle/vide et non « données de démonstration » (arbitrage AMENDEMENT-47, contraignant). Le gate d'âge 16+ (Apple 5.1.1), non prévu par la spec, est ajouté proprement au point de création de compte sans devenir un 5e écran obligatoire.

### RUN + BIKE (Partie II §2-§8, §18) — audit de conformité du code actuel

Le mode BIKE est ABSENT en bloc — aucune surface, aucun moteur, aucun flag, aucun terme i18n. Pire qu'une simple absence : l'architecture actuelle est structurellement ANTI-VÉLO.
